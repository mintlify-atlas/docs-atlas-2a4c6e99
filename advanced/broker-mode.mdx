## Overview

Broker mode provides policy-based permission control for AI agents, ensuring they can only perform authorized actions on your codebase.

<Warning>
  Experimental: AI agent governance is an emerging space. This feature is forward-looking and may change.
</Warning>

## Quick Start

<Steps>
  <Step title="Initialize Policy">
    ```bash
    connect broker init
    ```
    
    Creates `.connect/policy.yml` with secure defaults:
    ```yaml
    version: 1
    default: review
    
    rules:
      # Allow source code
      - path: src/**
        action: allow
      
      # Block sensitive files
      - path: .env*
        action: block
        reason: Environment files may contain secrets
    ```
  </Step>
  
  <Step title="Run AI Agent">
    ```bash
    # Run with broker enforcement
    connect broker aider
    
    # Or any AI agent
    connect broker run -- cursor
    connect broker run -- claude
    ```
  </Step>
  
  <Step title="Monitor Actions">
    ```bash
    # View audit log
    connect broker audit
    
    # View statistics
    connect broker audit --stats
    ```
  </Step>
</Steps>

## Policy Configuration

### Policy Structure

```yaml
version: 1
default: review  # allow | block | review

rules:
  # File access rules
  - path: src/**
    action: allow
  
  - path: .env*
    action: block
    reason: Environment files may contain secrets
  
  # Shell command rules
  - command: git add *
    action: allow
  
  - command: rm -rf *
    action: block
    reason: Destructive command
  
  # Git operation rules
  - git: force-push
    action: block
    reason: Force push can destroy history
```

### Actions

| Action | Behavior | Use Case |
|--------|----------|----------|
| `allow` | Permit silently | Source code, safe commands |
| `block` | Deny immediately | Secrets, destructive operations |
| `review` | Prompt for approval | Config files, git push |

### Pattern Matching

<AccordionGroup>
  <Accordion title="File Paths">
    Supports glob patterns:
    
    ```yaml
    # Match all TypeScript files
    - path: "**/*.ts"
      action: allow
    
    # Match specific directory
    - path: src/auth/**
      action: review
    
    # Match dotfiles
    - path: .env*
      action: block
    
    # Wildcards
    # * - matches any characters except /
    # ** - matches any characters including /
    # ? - matches single character
    # [abc] - matches any character in brackets
    ```
  </Accordion>
  
  <Accordion title="Commands">
    Token-based matching:
    
    ```yaml
    # Exact match
    - command: git status
      action: allow
    
    # Wildcard arguments
    - command: npm install *
      action: allow
    
    # Pattern matching
    - command: git push *
      action: review
    
    # Block dangerous patterns
    - command: rm -rf *
      action: block
    ```
    
    Security:
    - Commands with shell operators (`;`, `&&`, `|`) are blocked automatically
    - Prevents command chaining that could bypass policy
  </Accordion>
  
  <Accordion title="Git Operations">
    High-level git operation control:
    
    ```yaml
    # Git operations
    - git: commit
      action: allow
    
    - git: push
      action: review
    
    - git: force-push
      action: block
    
    - git: branch-delete
      action: review
    ```
  </Accordion>
</AccordionGroup>

### Execution Modes

<Tabs>
  <Tab title="Interactive (Default)">
    Prompts for approval on `review` actions:
    
    ```bash
    connect broker aider
    ```
    
    Output when agent attempts restricted action:
    ```
    ┌─────────────────────────────────────────────────┐
    │  🛡️  Agent Permission Request                    │
    ├─────────────────────────────────────────────────┤
    │  Type: file                                     │
    │  Target: .env.production                        │
    │  Reason: Environment files may contain secrets │
    └─────────────────────────────────────────────────┘
    
      Allow? [y/N]:
    ```
  </Tab>
  
  <Tab title="Auto-Approve">
    ```bash
    # CLI flag
    connect broker aider --yes
    
    # Environment variable
    CONNECT_AUTO_APPROVE=1 connect broker aider
    ```
    
    <Warning>
      Only use in trusted environments. All review actions will be approved.
    </Warning>
  </Tab>
  
  <Tab title="Auto-Deny">
    ```bash
    # CLI flag
    connect broker aider --no
    
    # Environment variable
    CONNECT_AUTO_DENY=1 connect broker aider
    ```
    
    Useful for CI/CD to ensure no unexpected actions.
  </Tab>
  
  <Tab title="Observe Only">
    ```bash
    connect broker aider --observe
    ```
    
    Logs actions but doesn't enforce. Useful for policy development.
  </Tab>
</Tabs>

## Git Hooks

### Installation

```bash
connect broker hooks
```

Installs hooks:
- **pre-commit**: Checks staged files against policy
- **pre-push**: Checks for force pushes

```bash
[ok] Installed 2 git hook(s)
  pre-commit: .git/hooks/pre-commit
  pre-push: .git/hooks/pre-push

  Existing hooks backed up and will be called after broker check.
  Run 'connect broker hooks --uninstall' to restore original hooks.
```

### Hook Behavior

<Steps>
  <Step title="pre-commit">
    Validates all staged files:
    
    ```bash
    git add .env.production
    git commit -m "Update env"
    ```
    
    Output:
    ```bash
    [x] Blocked: .env.production
      Environment files may contain secrets
    
    [x] Commit blocked due to policy violations
    ```
  </Step>
  
  <Step title="pre-push">
    Detects force pushes:
    
    ```bash
    git push --force
    ```
    
    Output:
    ```bash
    [x] Git force-push blocked by policy
      Reason: Force push can destroy history
    ```
  </Step>
</Steps>

### Hook Chaining

Existing hooks are preserved:

```bash
# Before: .git/hooks/pre-commit
#!/bin/sh
echo "Running tests..."
npm test

# After installation:
# Original hook backed up to .git/hooks/pre-commit.connect-backup
# New hook chains to original:

#!/bin/sh
# Connect Agent Permission Broker - pre-commit hook
connect broker git-check pre-commit || exit $?

# Chain to original hook
if [ -x ".git/hooks/pre-commit.connect-backup" ]; then
  ".git/hooks/pre-commit.connect-backup" "$@" || exit $?
fi
```

### Uninstallation

```bash
connect broker hooks --uninstall
```

Restores original hooks from backup.

## Security Features

### Shell Injection Prevention

Commands are validated for shell metacharacters:

```typescript
// Blocked patterns
const dangerousPatterns = [
  { pattern: /;\s*\S/, desc: 'command chaining (;)' },
  { pattern: /&&\s*\S/, desc: 'conditional chaining (&&)' },
  { pattern: /\|\|\s*\S/, desc: 'conditional chaining (||)' },
  { pattern: /\|\s*\S/, desc: 'piping (|)' },
  { pattern: /`[^`]+`/, desc: 'command substitution (backticks)' },
  { pattern: /\$\([^)]+\)/, desc: 'command substitution ($())' },
];
```

Example:

```bash
# Blocked automatically
connect broker exec -- 'ls; rm -rf /'

# Output:
[x] Command validation failed
  Command contains command chaining (;) which could bypass policy
```

### Workspace Boundaries

Prevents access outside project:

```typescript
const isOutsideWorkspace =
  relativePath === '..' ||
  relativePath.startsWith(`..${path.sep}`) ||
  path.isAbsolute(relativePath);

if (isOutsideWorkspace) {
  return {
    action: 'block',
    reason: 'Path is outside of the workspace boundary',
  };
}
```

## Troubleshooting

### Policy Not Found

```bash
[!] No policy found. Using secure defaults.
  Run 'connect broker init' to customize.
```

Solution:

```bash
cd /path/to/project
connect broker init
```

### Agent Bypass

Broker only controls agents run through `connect broker`:

```bash
# Protected
connect broker aider

# NOT protected
aider  # Direct execution
```

### Hook Not Triggering

```bash
# Verify hooks are installed
ls -la .git/hooks/

# Check permissions
chmod +x .git/hooks/pre-commit
chmod +x .git/hooks/pre-push

# Reinstall
connect broker hooks
```
